Files We Will Build

File Responsibility
config.pyAll constants — model names, paths, chunk sizes, top-k values
chunker.pyHierarchical chunking — parent sections + child paragraphs, each child stores parent_id
ingest.pyOne-time: chunk → embed children → ChromaDB + BM25 index saved to disk
retriever.pyDense + BM25 → RRF merge → cross-encoder rerank → parent lookup
generator.pyGPT-4o: question + 5 parent sections → grounded answer
evaluator.pyRAGAS: retrieval recall + faithfulness + answer correctness vs golden dataset
logger.pyConsole (human-readable) + JSON file (full trace) per query
query.pyEntry point: python3 query.py "your question"
logs/JSON log files, one per run, named by timestamp
🧪 tests/test_chunker.pyUnit tests: child-parent links correct, section boundaries respected, no empty chunks
🧪 tests/test_retriever.pyUnit tests: RRF merge ranks correctly, handles zero results from one source

Testing Philosophy — What We Test and Why

✅ Unit test these
  • Chunker hierarchy — pure logic, deterministic, everything depends on it being correct
  • RRF merge algorithm — pure math, no API calls, easy to verify with known inputs
❌ Don't mock these
  • Reranker — mocking gives no signal about actual relevance scoring
  • ChromaDB — mock retrieval tells you nothing about real embedding quality
  • GPT-4o — mock answers can't test faithfulness or correctness
🎯 The real integration test

Running all 20 golden dataset questions through the full pipeline and measuring RAGAS scores. This catches real failures that mocks never would.

Full Query Pipeline (unchanged)

python3 query.py "What was Tesla's revenue in 2024?"
  → Dense search: ChromaDB top-20 by cosine similarity
  → BM25 search: keyword match top-20
  → RRF merge: combine both → 40 candidates
  → Reranker: cross-encoder → keeps top-5
  → Parent lookup: child chunks → full parent sections
  → GPT-4o: question + 5 parent sections → answer
  → RAGAS: retrieval + faithfulness + correctness
  → Logger: console + logs/ JSON
